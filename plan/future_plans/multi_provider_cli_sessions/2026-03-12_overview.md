# Multi-Provider CLI Session Runtime Overview

## Purpose

Preserve and sharpen a future direction where primary orchestration sessions can run through multiple external CLI coding tools while remaining durably inspectable and directly recoverable.

This is not an implementation plan.

It is a working note for a later runtime expansion.

## Current Repo Reality

The review of the current repo shows four important facts.

### 1. Fresh primary-session bootstrap is Codex-specific

Today the primary-session launch path builds only Codex bootstrap commands:

- `src/aicoding/daemon/session_manager.py` builds `codex_fresh` and `codex_resume_last` launch modes
- `src/aicoding/daemon/codex_session_bootstrap.py` is the only shipped provider bootstrap helper

There is no provider registry, provider capability model, or provider-specific launcher family for Gemini or Claude yet.

### 2. Recovery replacement still assumes Codex `resume --last`

The current recovery bootstrap for replacement sessions is:

- `PYTHONPATH=src python3 -m aicoding.daemon.codex_session_bootstrap resume`

That helper then executes:

- `codex --yolo resume --last`

This means durable recovery still depends on the working-directory-local "last session" heuristic instead of a provider session handle that the daemon captured and can target directly.

### 3. Durable `provider_session_id` is not true provider identity yet

The current bind and replacement flows in `src/aicoding/daemon/session_records.py` store:

- `provider = adapter.backend_name`
- `provider_session_id = adapter_snapshot.session_name`

In the current tmux-backed setup, that means `provider_session_id` is effectively the tmux session name, not a provider-owned Codex, Gemini, or Claude session identifier.

So the database already has the right column name, but not yet the right meaning for multi-provider recovery.

### 4. Provider-aware recovery surfaces exist, but the provider contract is still thin

The repo already has provider-aware recovery planning and status surfaces:

- `plan/features/17_F34_provider_agnostic_session_recovery.md`
- `plan/features/67_F12_provider_specific_session_recovery_surface.md`
- `src/aicoding/daemon/session_records.py` provider recovery snapshots

That is useful groundwork, but it does not yet prove true provider restore semantics because the underlying stored identity is still tmux-oriented.

## Desired Future Direction

The future system should separate three concerns that are currently collapsed together.

### Transport session

This is the tmux or terminal wrapper used for attachability, liveness checks, and operator inspection.

### Provider runtime

This is the external CLI tool actually doing the work:

- Codex
- Gemini
- Claude
- potentially other tools later

### Provider session identity

This is the provider-owned identifier or restoration token that can be used to resume the exact same session directly when the provider supports it.

The durable model should not assume these three identities are the same.

## Main Planning Goal

The strongest version of this future feature is not just "let the launcher pick a different executable."

It is:

- choose a provider-specific bootstrap strategy
- capture any provider-specific session identifier durably
- expose provider-aware inspection and recovery surfaces
- fall back safely when a provider does not offer direct resume semantics

That matters because direct session recovery is one of the biggest practical value points of long-lived orchestration.

## Suggested Future Architecture Shape

The eventual runtime likely needs a provider integration layer with code-owned capabilities such as:

- provider name
- fresh-launch command builder
- replacement-launch command builder
- session identity extraction strategy
- prompt bootstrap strategy
- workspace-trust or consent handshake rules
- direct resume support
- attach or inspect support
- unsupported-feature fallback rules

One reasonable direction would be a provider registry with entries like:

- `codex`
- `gemini`
- `claude`

Each provider entry would declare whether it supports:

- fresh prompt-driven bootstrap
- direct resume by explicit session id
- resume only by cwd-scoped last-session lookup
- no provider restore at all
- machine-readable status or transcript extraction

## Session Identity Capture Requirement

The current Codex behavior suggests an immediate planning target:

- run a provider-native status or metadata command after bootstrap
- parse the returned provider session id
- record that id durably as distinct from tmux session identity

For Codex, the user specifically called out a future direction like:

- run `/status`
- extract the session UUID
- record it durably
- later resume that specific session directly instead of assuming `--last`

The equivalent discovery work still needs to be done for Gemini and Claude.

This future plan should assume the exact session-identity capture contract is provider-specific until proven otherwise.

## What Must Be Learned Before Implementation

The highest-value discovery questions are:

1. Does Codex expose a stable machine-parseable session id through `/status`, CLI flags, files, or another surface?
2. Does Codex support direct resume by explicit session id in the current CLI version?
3. What equivalent session or conversation identity surfaces exist for Gemini CLI?
4. What equivalent session or conversation identity surfaces exist for Claude CLI?
5. Which providers can restore the same session versus only reopen the same cwd context?
6. Which provider surfaces are safe to call non-interactively from tmux bootstrap and daemon recovery flows?

Without answers to those questions, any provider-agnostic abstraction would be guesswork.

## Six-System Impact If This Becomes Real Work

### Database

Would likely need durable separation between:

- tmux transport identity
- provider name
- provider session id
- provider restoration metadata
- provider capability or recovery evidence recorded at bind time

### CLI

Would likely need explicit commands for:

- showing provider session identity
- showing provider recovery status
- forcing provider-specific rebind or fallback
- capturing or refreshing provider session metadata

### Daemon

Would likely own:

- provider registry selection
- provider bootstrap execution
- provider status/identity capture
- provider-aware replacement and resume decisions
- fallback from provider-aware recovery to provider-agnostic recovery

### Website UI

Would likely need session detail surfaces that distinguish:

- tmux session name
- provider
- provider session id
- recovery posture
- why direct restore is or is not available

### YAML

Probably should stay minimal.

The safest current assumption is:

- provider selection may become a policy input
- provider bootstrap, capture, and recovery mechanics stay code-owned

### Prompts

Bootstrap and recovery instructions likely become provider-specific, especially when:

- the provider needs a status command after startup
- the provider has different trust prompts
- the provider needs different resume instructions

## Key Invariants Worth Preserving

- The database remains authoritative over session ownership and history.
- tmux remains a transport and inspectability layer, not the durable source of provider truth.
- provider session identity, when available, must be stored separately from tmux session identity.
- recovery may use provider-specific restore when available, but must keep a provider-agnostic fallback.
- unsupported provider restore semantics must be explicit, not silently guessed.
- the daemon must record how a provider session id was obtained and whether it is verified, inferred, or unavailable.

## Most Likely Narrow First Slice

If this ever becomes active work, the safest first slice is probably:

1. Refactor the current Codex-only launch path behind a provider interface while keeping Codex as the only implemented backend.
2. Add durable separation between tmux identity and provider session identity.
3. Teach the Codex backend to capture and persist a direct session identifier if a reliable surface exists.
4. Only after that discovery succeeds, add one more provider, probably the CLI with the cleanest non-interactive restore story.

That would avoid inventing a fake provider abstraction before the first real identity-capture contract exists.
