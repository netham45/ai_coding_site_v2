# Stage 05: Post-V1 Evolution

## Purpose

Post-v1 work is choice-driven rather than purely linear.

After the repository earns a first flow-complete or release-ready scope, later work should still reopen discovery, architecture, setup, feature, hardening, or migration disciplines intentionally.

## Named Workstreams

- major feature expansion
- architecture or subsystem overhaul
- assurance, audit, or compliance pass
- migration or offload
- deprecation, archival, or sunset

## Required Outcomes

- the chosen workstream is named explicitly
- reopened lifecycle obligations are identified
- new proof requirements are written down
- baseline maturity and active workstream are both visible in operational-state tracking

## Common Failure Modes

- treating post-v1 work as generic maintenance
- hiding major architectural changes inside ordinary feature language
- reopening risk without reopening proof expectations
- losing sight of already-earned baseline maturity

## Exit Condition

This stage is governed correctly when the repository can evolve materially without regressing into vague, undocumented maintenance work.
